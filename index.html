Arctic Observing Viewer

About

"AOV was born from a recognition that various Arctic Observing networks needed a tool – beyond project tracking systems and data catalogs – to strategically assess status and progress for long-term monitoring programs. A new, intermediate level of “granularity” (between project-level information and dataset-level information) was needed in a portal that focused on locations, activities, and resources. These are encapsulated in this mapping application as “observing sites.” AOV is evolving and growing, founded on the principle that collaborative information sharing can provide a comprehensive – and therefore useful – perspective." - arcticobservingviewer.org

Vision

"Arctic Observing – spread as it is among various national and international initiatives – could benefit from an improved cyber-infrastructure that facilitates further integration, discovery, and analysis between funding bodies, PIs, data centers, users, etc." - arcticobservingviewer.org

The ABoVE Spectral Library (ASTRAL)

About

"NASA’s Terrestrial Ecology Program launched the Arctic-Boreal Vulnerability Experiment (ABoVE) in an effort to better understand the vulnerability and resilience of Arctic and Boreal ecosystems and societies to environmental variability and change. Remote sensing of natural targets, in particular hyperspectral reflectance, from ground, airborne and space borne platforms is a widely used method for monitoring changing arctic landscapes, but remains poorly documented for much of the Arctic. In an effort to alleviate these issues and help bring together our understanding, we have developed a hyperspectral data library fused with a web mapping application that allows for the discovery, visualization, and sharing of spectral data across the ABoVE domain." - The ABoVE Spectral Library (ASTRAL)

Vision

"During the 2017-2019 field seasons, our project alone collected over 3,500 field spectra at the leaf and plot level for validation of AVIRIS NG spectral reflectance and we have sourced more than 5,000 spectra from our archives and those of others that would likely be useful for ABoVE efforts. These include web-based spectral libraries and informational tools such as pre-existing BAID- the Barrow Area Information Database, AOV- the Arctic Observing Viewer, EcoSIS, ASU Spectral Library, JPL HyspIRI Spectral Library, the USGS Digital Spectral Library, and the ASTER Spectral Library. " - The ABoVE Spectral Library (ASTRAL)

The Irrigation Web Application

About

To get an approach to the real irrigation system and advantages it brings to farmers, the irrigation app wants to keep track and get better the crop wellness. By using different technology, our main purpose is to develop an application to estimate the actual evapotranspiration (ETa) to determine the water needs and determine whether a crop would need more water or not. The department of computer science along with the geology department at the University of Texas at El Paso wants to develop a useful tool for different areas in our community. Irrigation is the process of delivering water to plants so as to meet their needs for several important resources. (Lieth and Oki., 2019 10.1016/B978-0-444-63696-6.00009-8 Chapter 9 - Irrigation in Soilless Production.) To get the desired information about a certain crop, we are using Google Earth Engine. It provides us with satellite imagery and geospatial datasets with planetary-scale analysis. We use these satellite images to detect changes in the crops.

Scope

Our purpose is to calculate the Actual Evapotranspiration (ETa) through Reference Evapotranspiration (ET0) and Crop coefficient (Kc) as follows; ETa =ET0*Kc. By knowing the ET0 and Kc, we can estimate how much water a crop has. We can imagine it as a bucket with water. Whenever the bucket begins to empty, we know that the crop needs water by the irrigation process. There are different irrigation systems, and most of them use electricity to provide water to the crop. The irrigation app looks for a better approach to maximize the income of farmers by using wisely the resources such as water and energy. We can get a better reaching of the crop quality by knowing some extra parameters like the type of soil, irrigation method, planting date, hourly rate, and the hydraulic design of irrigation systems. Our main target will be farmers. The use of this app by farmers will enable the huge advantage of knowing the status of their crop and many other details

My Contribution

The irrigation web application is divided into front-end and back-end. Using HTML, JavaScript, and CSS in front-end applications and python in the back-end server. Using the ArcGIS API, the app is rendering a map that is one of the main characteristics. It gets all the libraries required to render all the tools needed in our app:

  • Map
  • MapView
  • CoordinateConversion
  • Graphic
  • Sketch
  • GraphicsLayer
  • dom

JavaScript uses the “Map” object to specify which base map image is going to use. “MapView” renders the map in a div HTML tag. The view object has as attributes the container that is the div in which the map will be rendered. The map attribute takes the before map Object created. It sets a zoom of the map view and a map center. The next object used is the sketch object. The sketch is used to draw a polygon on the map to get all the information on it. The sketch object has as attributes the layer, view, creation mode, and the available create tools. To find the coordinates within the map, I use a function called show coordinates. This function gets the latitude and longitude values from the map to then put them together in a widget.

Medical Scheduler

Description

A certain medical office does home visits for housebound patients. Scheduling the visits efficiently is important, and I will build a prototype web-based system to help. Every afternoon the scheduler starts with a list of patients to be visited tomorrow. For each patient, she has the mane, whether or not this is a new patient, and sometimes time constraints. For example, if the patient’s daughter needs to be there but is only available in the anteroom. There is also a patient master list, which contains names, addresses, approximate map coordinates, etc. the output is a tentative schedule, which she can edit if necessary, and then give to the clinician who will do the home visits. The goal of assignment 1 is to learn how PHP works and what its features are.
